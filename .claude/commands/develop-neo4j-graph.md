# Develop Neo4j Graph — End-to-End Workflow

Develop Neo4j graphs end-to-end: analyze source data, design data models, ingest data, and validate with Cypher queries. Use when importing data to Neo4j, designing graph data models, creating knowledge graphs, or working with Neo4j MCP servers.

## Progress Checklist

Copy this checklist to track progress:

```
- [ ] Step 1: Run discovery on sample of source data
- [ ] Step 2: Iteratively design the graph data model
- [ ] Step 3: Ingest source data according to finalized data model
- [ ] Step 4: Generate Cypher according to use cases
- [ ] Step 5: Validate that graph adequately addresses use cases
- [ ] Step 6: Generate final report
```

---

## Step 1: Discovery

### Structured data (CSV)

- Read samples of source data provided by the user. Do not persist entire datasets unless they are under 100 rows.
- Note there may be nested folders in the `data/` directory.
- Analyze the source data (read CSV files, summarize schema, identify key columns).
- Assess whether the use cases provided by the user apply to the source data.
- Insights will inform the graph data modeling process.

### Unstructured data (PDF)

- Sample data should be in `data/` (do not persist entire datasets unless under 10 items).
- Identify entities and relationships that exist throughout the documents and pertain to the provided use cases.

---

## Step 2: Graph Data Modeling

Use the `neo4j-data-modeling` MCP server. Follow this process:

### Primary Instructions
- If you know the source information for Properties, include it in the data model.
- If you deviate from the user's requests, clearly explain why.
- Only use data from the provided sample data to create the data model (unless explicitly stated otherwise).
- If the user requests use cases outside the scope of the provided sample data, explain why you cannot model those use cases.

### Process

**1. Analysis**
- Analyze the sample data
- Use `list_example_data_models` to check for relevant examples
- Use `get_example_data_model` to retrieve any relevant examples

**2. Generation**
- Generate a new data model based on your analysis, context, and examples
- Use `get_mermaid_config_str` to validate the model and get a Mermaid visualization
- Correct any validation errors and repeat

**3. Final Response**
- Show the user a Mermaid visualization (if supported)
- Explain the data model and any gaps between requested use cases
- Request feedback (data modeling is iterative)

### Refinement
- Prompt the user for feedback on the graph data model
- Make necessary changes and repeat step 2

### Final Data Model
- When no more changes are needed, persist the data model as JSON in `outputs/data_models/`

---

## Step 3: Ingest Data

### Structured data (CSV) — use `neo4j-ingest`

- Use the Ingest MCP Server (`neo4j-ingest`) to load data
- Each node and relationship ingest query should be generated by the data modeling MCP server
- Note: each CSV file may be ingested more than once if it contains multiple node or relationship types
- Always provide **absolute paths** to CSV files (not relative paths)

### Unstructured data (PDF) — use `neo4j-lexical-graph` then `neo4j-entity-graph`

**Lexical graph phase** (`neo4j-lexical-graph`):
1. Use `create_lexical_graph_from_pdf` or `create_lexical_graph_from_folder` to parse PDFs
2. This creates `Document` and `Chunk` nodes in Neo4j
3. Use `create_chunk_embeddings` to generate vector embeddings (requires LLM API key)
4. Use `create_fulltext_index` for keyword search capability

Parse modes:
- `pymupdf` — default, fast, good for text-heavy PDFs
- `docling` — structural layout detection
- `page_image` — vision model per page
- `vlm_blocks` — vision model per layout block (highest quality, slowest)

**Entity graph phase** (`neo4j-entity-graph`):
1. Use `extract_entities_from_chunks` to extract structured entities from Chunk nodes
2. Processing is async — use `get_entity_extraction_status` to monitor progress
3. Entities become first-class nodes in the graph, linked to their source chunks

## Step 4: Generate Cypher for Use Cases

- Each use case should have one or more Cypher queries that provide appropriate context
- Persist Cypher queries as a YAML file in `outputs/queries/` using this format:

### YAML Format

```yaml
analytical_queries:
    use_case_key:
        name:
        cyphers:
```

### Example

```yaml
analytical_queries:
  apple_count:
    name: How many apples are in the basket?
    cyphers:
      - MATCH (a:Apple)-[:IN_BASKET]->(b:Basket {id: $basketId}) RETURN COUNT(*)

  apple_and_basket_counts:
    name: How many apples and baskets are there?
    cyphers:
      - |
        MATCH (a:Apple)
        WITH DISTINCT a
        RETURN COUNT(*) as apple_count
      - |
        MATCH (b:Basket)
        WITH DISTINCT b
        RETURN COUNT(*) as basket_count
```

---

## Step 5: Validate

- Use `neo4j-graphrag` (`read_neo4j_cypher`) to execute the analytical Cypher queries
- Analyze results and ensure they address the original use cases
- Note any gaps or issues

---

## Step 6: Generate Report

Present a final report to the user including:

1. **Source data** — description of each input source and its characteristics
2. **Use cases** — the goals defined during discovery
3. **Graph data model** — final model with descriptions of each node label and relationship type
4. **Cypher queries** — each analytical query with a description of what it answers
5. **Gaps** — anything not addressed, due to:
   - Missing source data
   - Inadequate Cypher
   - Unclear use cases
   - Other reasons
6. **Summary** — current state of the application and recommended next steps

Persist the report as a Markdown file in `outputs/reports/`.

## MCP Server Quick Reference

| Server | Purpose | Key Tools |
|--------|---------|-----------|
| `neo4j-data-modeling` | Schema design | `list_example_data_models`, `get_example_data_model`, `get_mermaid_config_str` |
| `neo4j-ingest` | CSV loading | `ingest_csv_to_neo4j` |
| `neo4j-lexical-graph` | PDF → graph | `create_lexical_graph_from_pdf`, `create_chunk_embeddings`, `create_fulltext_index` |
| `neo4j-entity-graph` | Entity extraction | `extract_entities_from_chunks`, `get_entity_extraction_status` |
| `neo4j-graphrag` | Read/write Cypher + RAG retrieval | `read_neo4j_cypher`, `write_neo4j_cypher`, `get_neo4j_schema_and_indexes`, `vector_search`, `fulltext_search`, `search_cypher_query` |
